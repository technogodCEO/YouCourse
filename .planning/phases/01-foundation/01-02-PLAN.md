  <action>
    Create `src/actions/auth.ts` with `'use server'` at the top. Three exported actions:

    `signup(state: FormState, formData: FormData)`:
    1. Parse with `SignupSchema.safeParse({ email: formData.get("email"), password: formData.get("password") })`. On failure return `{ errors: validated.error.flatten().fieldErrors }`.
    2. Hash: `const hashed = await bcrypt.hash(password, 12)` (import `bcrypt from "bcryptjs"`).
    3. Insert into users MANUALLY before signIn (Pitfall 2 — adapter does not persist passwords): `await db.insert(users).values({ email, password: hashed })`. Wrap in try/catch; on unique-violation return `{ message: "An account with this email already exists. Sign in instead?" }`.
    4. Then `await signIn("credentials", { email, password, redirectTo: "/dashboard" })`. Note: signIn throws a redirect — do NOT wrap the signIn call in the same try/catch as the insert, or the redirect will be swallowed. Let the redirect propagate.

    `login(state: FormState, formData: FormData)`:
    1. Parse with `LoginSchema.safeParse(...)`. On failure return `{ errors: ... }`.
    2. Call `await signIn("credentials", { email, password, redirectTo: "/dashboard" })` inside a try/catch that re-throws redirect errors. On `AuthError`/`CredentialsSignin` (invalid creds), return `{ message: "Incorrect email or password. Double-check your details and try again." }`. Import `AuthError` from `"next-auth"`; structure as `if (error instanceof AuthError) return {...}; throw error` so the Next.js redirect re-throws.

    `logout()`:
    - `'use server'` action: `await signOut({ redirectTo: "/login" })`. Import `signOut` from `@/lib/auth`.

    Concrete imports at top:
    ```typescript
    'use server'
    import { db } from "@/lib/db"
    import { users } from "@/lib/db/schema"
    import { signIn, signOut } from "@/lib/auth"
    import { SignupSchema, LoginSchema, type FormState } from "@/lib/definitions"
    import bcrypt from "bcryptjs"
    import { AuthError } from "next-auth"
    ```
  </action>

    Create `proxy.ts` at PROJECT ROOT (NOT middleware.ts — Pitfall 1; Next.js 16 name per D-08). Implement D-09 route lists and D-10 redirect-to-login. The proxy must read the Auth.js session. Two valid approaches — pick the one matching the Auth.js cookie:

    Preferred (use Auth.js `auth` wrapper, simplest and correct cookie handling):
    ```typescript
    import { auth } from "@/lib/auth"
    import { NextResponse } from "next/server"

    const protectedRoutes = ["/dashboard", "/courses", "/create"]   // D-09
    const publicRoutes = ["/", "/login", "/signup", "/forgot-password", "/reset-password"]

    export default auth((req) => {
      const path = req.nextUrl.pathname
      const isProtected = protectedRoutes.some(r => path.startsWith(r))
      const isLoggedIn = !!req.auth?.user
      if (isProtected && !isLoggedIn) {
        return NextResponse.redirect(new URL("/login", req.nextUrl))   // D-10
      }
      if ((path === "/login" || path === "/signup") && isLoggedIn) {
        return NextResponse.redirect(new URL("/dashboard", req.nextUrl))
      }
      return NextResponse.next()
    })

    export const config = {
      matcher: ["/((?!api|_next/static|_next/image|favicon.ico|.*\\.png$).*)"],
    }
    ```
    The matcher MUST skip `api`, `_next/static`, `_next/image`, and `.png` assets (never run on the Auth.js API route or static logo). Guard the authenticated->public redirect to ONLY `/login` and `/signup` (not all public routes) to avoid the redirect loop in Pitfall 5.

    If `auth((req) => ...)` import causes an Edge bundling error from the Drizzle adapter (adapter is not edge-safe), fall back to the cookie-decrypt approach from RESEARCH §Pattern 3 using `decrypt()` and the Auth.js session cookie name (`authjs.session-token` / `__Secure-authjs.session-token`). Document whichever approach was used in the SUMMARY. (The `publicRoutes` array is kept for reference/documentation of D-09 even when only /login and /signup trigger the reverse redirect.)
